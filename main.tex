% ABSTRACT
% ============================================================
\begin{abstract}
Large Language Models (LLMs) demonstrate remarkable capabilities in multi-step reasoning, yet existing prompting methods such as Chain-of-Thought (CoT) lack explicit mechanisms for managing assumptions, constraints, and validation criteria. We introduce PWBS (Prompt Work Breakdown Structure), a structured planning framework inspired by Work Breakdown Structure (WBS) that organizes LLM prompts into a 6-tuple: Goal, Assumption, Formula, Constraint, Task, and Validation (G-A-F-C-T-V). \textbf{PWBS targets multi-constraint structured-output tasks}---such as travel planning, code debugging, and data analysis---where multiple conditions must be simultaneously satisfied and outputs require complex structure, rather than single-answer problem solving.

Through experiments on 75 self-designed benchmark tasks across three categories (travel planning, code debugging, logical reasoning) with GPT-4o and GPT-4o-mini, we find that PWBS does not uniformly outperform baselines in overall task success rate---no statistically significant differences were observed ($p > 0.05$). However, component-level ablation reveals that \textbf{Assumption surfacing provides a consistent +4.0\%p improvement on both models}, making it the most robust individual contribution. Furthermore, PWBS components show \textbf{inverse scaling with model capability}: Constraint specification yields +8.0\%p on GPT-4o-mini but $-$2.7\%p on GPT-4o, suggesting that structured prompting is most beneficial for weaker models. Validation on public benchmarks (GSM8K \citep{cobbe2021gsm8k}, BIG-Bench Hard \citep{suzgun2023challenging}) confirms this scope: PWBS underperforms baselines on single-answer tasks ($-$27\%p overall) but achieves the highest score on \texttt{boolean\_expressions} (96.7\%), where logical structure aligns with PWBS components.

Additional contributions include: (1) a \textbf{Reasoning Consistency} improvement of +32.0\%p in code debugging tasks, (2) a \textbf{CSR Transparency Paradox} revealing that PWBS's lower Constraint Satisfaction Rate reflects better constraint detectability rather than worse performance, (3) \textbf{Adaptive Tolerance} for systematic constraint relaxation, and (4) a \textbf{Failure-Driven Evolution} narrative documenting how the framework evolved from 5-tuple to 6-tuple through iterative failure analysis.

These findings reposition PWBS from a monolithic framework to an \textbf{adaptive toolkit} where practitioners should selectively apply components based on model capability and task characteristics.
\end{abstract}

\noindent\textbf{Keywords:} Prompt Engineering, Structured Planning, Work Breakdown Structure, LLM Agents, Chain-of-Thought

\section{Introduction}

\subsection{Problem Definition}

Large Language Models (LLMs) have shown remarkable performance across diverse multi-step reasoning tasks. However, structural limitations persist in LLM reasoning: premises of reasoning remain implicit, constraints are not systematically managed, and verification criteria for results are absent. This leads to plausible but incorrect reasoning (hallucination), logical inconsistency between steps, and unstable outputs for identical tasks.

Chain-of-Thought (CoT) prompting \citep{wei2022chain} significantly improved reasoning quality through the instruction ``let's think step by step.'' However, CoT does not provide explicit structure for what premises each step is based on, what constraints must be satisfied, or how to verify the final result. Plan-and-Solve \citep{wang2023plan} introduced a plan-then-execute structure but similarly lacks explicit mechanisms for managing assumptions and constraints.

\subsection{Research Motivation: Inspiration from WBS}

In project management, Work Breakdown Structure (WBS) is a foundational tool defined by PMBOK \citep{pmi2021pmbok} that hierarchically decomposes project scope into manageable deliverables to prevent omission and duplication. PWBS (Prompt Work Breakdown Structure) draws on two specific WBS principles: (1) \emph{scope completeness}---WBS ensures all deliverables are identified before execution begins, analogous to PWBS's Assumption and Constraint surfacing before reasoning; and (2) \emph{explicit decomposition}---WBS makes implicit project scope visible, analogous to PWBS making implicit reasoning premises visible. We note that the analogy is partial: WBS decomposes \emph{deliverables} hierarchically, while PWBS decomposes \emph{reasoning prerequisites} along semantic dimensions (G-A-F-C-T-V). The 6-tuple structure is closer to a requirements specification than a hierarchical work package tree; the WBS inspiration lies in the \emph{principle} of making implicit scope explicit, not in structural isomorphism.

\subsection{Research Questions}

This study is designed to answer the following questions:

\begin{itemize}[leftmargin=2em]
  \item \textbf{RQ1:} Do PWBS's structural components improve LLM reasoning quality? Specifically, which components contribute the most?
  \item \textbf{RQ2:} How does PWBS's effectiveness vary with model capability?
  \item \textbf{RQ3:} Does PWBS improve reasoning consistency?
  \item \textbf{RQ4:} Is the effectiveness maintained when AI automatically generates the 6-tuple structure?
\end{itemize}

\subsection{Key Findings (Summary)}

Experiments on 75 self-designed benchmark tasks showed that PWBS did not demonstrate statistically significant differences in overall success rate compared to existing methods ($p > 0.05$). However, component-level analysis revealed the following key findings:

\begin{enumerate}[leftmargin=2em]
  \item \textbf{Assumption surfacing has a model-agnostic effect.} Adding only Assumption to CoT yields +4.0\%p improvement on both GPT-4o and GPT-4o-mini. This means explicit articulation of implicit assumptions improves reasoning quality regardless of model capability.

  \item \textbf{PWBS's effectiveness inversely correlates with model capability.} Constraint specification shows +8.0\%p effect on GPT-4o-mini but $-$2.7\%p on GPT-4o. Stronger models already implicitly recognize constraints, so explicit structure disperses attention.

  \item \textbf{Component accumulation shows diminishing returns.} Full PWBS (CoT+ACFV) shows lower performance than optimal subsets (CoT+A or CoT+AC). This suggests PWBS should be applied selectively rather than uniformly.

  \item \textbf{PWBS dramatically improves reasoning consistency in code debugging.} Reasoning Consistency improves from CoT 2.7\% to PWBS 34.7\% (+32.0\%p).

  \item \textbf{CSR Transparency Paradox:} PWBS's low Constraint Satisfaction Rate (48.7\% vs Direct 84.3\%) is not performance degradation but a result of improved transparency in detecting constraint violations.
\end{enumerate}

\subsection{Contributions}

The contributions of this study are:

\begin{enumerate}[leftmargin=2em]
  \item \textbf{PWBS Framework:} We define a 6-tuple structure (G-A-F-C-T-V) and present a systematic component classification criterion through the 4-stage Invalidation Test.

  \item \textbf{Adaptive Mechanisms:} We introduce engineering mechanisms---Adaptive Tolerance, Formula Preserving, Data Preserving, and Constraint Coupling---developed to address failures discovered during experiments.

  \item \textbf{Component-level Effect Analysis:} We propose an ablation design that measures each component's independent and combinatorial contribution on top of CoT, rather than comparing the entire framework.

  \item \textbf{Transparent Record of Failure-Driven Evolution:} We document PWBS's evolution from 5-tuple to 6-tuple with failure cases, enhancing reproducibility and credibility.

  \item \textbf{Repositioning as Adaptive Toolkit:} We propose PWBS as an adaptive toolkit for selective component application based on model capability and task characteristics.
\end{enumerate}

\section{Related Work}

\subsection{Prompting Methodologies}

\textbf{Chain-of-Thought (CoT).} \citet{wei2022chain} demonstrated that the simple instruction ``Let's think step by step'' can significantly improve LLM multi-step reasoning performance. CoT generates explicit intermediate reasoning steps, guiding LLMs to solve complex problems incrementally. However, CoT lacks structure for the premises, constraints, and verification of each step, which can lead to logical inconsistency between steps.

\textbf{Self-Consistency.} \citet{wang2023selfconsistency} improved CoT by sampling multiple reasoning paths and selecting the most consistent answer via majority voting. Our Reasoning Consistency metric is related but distinct: Self-Consistency is a \emph{decoding strategy} that aggregates diverse reasoning paths at inference time, whereas our RC metric \emph{measures} the structural stability that a prompting framework induces across independent runs. PWBS's consistency improvement (+32.0\%p in code debugging) comes from structural scaffolding that constrains reasoning paths, not from output aggregation.

\textbf{Plan-and-Solve.} \citet{wang2023plan} proposed a two-phase approach: ``First understand the problem and devise a plan, then solve step by step.'' This introduces a higher-level planning structure beyond CoT's simple step enumeration. However, Plan-and-Solve also lacks mechanisms for explicit assumption management, systematic constraint handling, and result verification.

\textbf{Task Decomposition Approaches.} Least-to-Most prompting \citep{zhou2023leasttomost} decomposes complex problems into progressively easier subproblems, while Decomposed Prompting \citep{khot2023decomposed} delegates subtasks to specialized sub-prompts. PWBS shares the decomposition motivation but differs in two key ways: (1) PWBS decomposes along \emph{semantic dimensions} (assumptions, constraints, formulas) rather than problem difficulty or subtask specialization, and (2) PWBS's Constraint Coupling mechanism (Section~3.7) explicitly manages inter-subtask dependencies, addressing over-decomposition failures that naive decomposition approaches do not handle.

\textbf{Tree of Thoughts (ToT).} \citet{yao2024tree} proposed exploring reasoning paths in a tree structure, overcoming CoT's limitation of being confined to a single path. However, ToT focuses on search efficiency and does not address assumption or constraint management at each node.

\textbf{ReAct and Reflexion.} ReAct \citep{yao2023react} proposed an agent paradigm alternating reasoning and action, while Reflexion \citep{shinn2023reflexion} proposed iterative improvement through self-reflection. These share similar motivation to PWBS's Validation in utilizing execution feedback, but do not include assumption surfacing or structural constraint management.

\textbf{Program of Thoughts (PoT).} \citet{chen2023program} disentangles computation from reasoning by generating executable programs. PWBS's Formula component serves a related purpose---preserving computational procedures verbatim---but operates at the prompt-structuring level rather than requiring code generation, making it applicable to non-numerical tasks such as travel planning.

\subsection{Structured Prompting}

Recent research has attempted to improve LLM performance by adding structure to prompts. \citet{schulhoff2024prompt} provide a comprehensive survey of prompting techniques, documenting the growing evidence that structural elements in prompts improve LLM performance. Skeleton-of-Thought \citep{ning2024skeleton} structures outputs via skeleton-then-expansion for efficient parallel generation, focusing on output organization rather than input premises.

PWBS extends this structuring approach but differentiates itself by providing \emph{semantic} structuring based on engineering principles (WBS) rather than mere formatting or output organization. Specifically, PWBS structures the \emph{input} premises (assumptions, constraints, formulas) that govern reasoning, complementing methods that structure the reasoning \emph{process} (CoT) or \emph{output} (Skeleton-of-Thought).

\section{PWBS Framework}

\subsection{Overview}

PWBS is a structural planning framework that applies WBS principles from software engineering and project management to LLM agent prompt design. Just as WBS hierarchically decomposes project scope to prevent omission and duplication, PWBS decomposes tasks into explicit components to ensure reasoning completeness and consistency.

\textbf{PWBS does not replace CoT but functions as a meta-structure that strengthens CoT by adding a structuring layer on top of it.}

\subsection{Formal Definition}

PWBS is formally defined as a 6-tuple:
\begin{equation}
\text{PWBS} = (G, A, F, C, T, V)
\end{equation}

\noindent where each component is defined as follows:

\textbf{G (Goal):} Clearly describes the ultimate objective. Must be measurable and unambiguous, including original data/code.

\textbf{A (Assumption):} Set of premises the LLM must assume to be true. This is PWBS's core contribution---the most overlooked element in existing prompting methods. LLMs tend to create implicit assumptions, and when unstated, these become primary sources of hallucination and reasoning errors. Experiments confirmed Assumption surfacing as the only component showing consistent +4.0\%p improvement regardless of model capability.

\textbf{F (Formula):} Preserves calculation formulas verbatim from the task. Distinguished from Constraints---Constraints are boundary conditions judging solution eligibility (e.g., ``budget $\leq$ \$5,000''), while Formulas are calculation rules that must be applied exactly (e.g., ``score = exam$\times$0.4 + interview$\times$0.4 + service/10$\times$0.1 + cert$\times$5$\times$0.1''). This component was introduced after pilot experiments revealed formula abstraction failures (see Section~\ref{sec:evolution}, Phase~3).

\textbf{C (Constraint):} Conditions that must be satisfied during task execution. Includes time limits, resource limits, format requirements, and prohibitions. However, experiments show Constraint specification is only effective for weaker models (+8.0\%p on GPT-4o-mini) and can be counterproductive for stronger models ($-$2.7\%p on GPT-4o).

\textbf{T (Task):} Ordered set of subtasks. Each Task has three sub-elements: \textit{why} (rationale), \textit{method} (specific approach), \textit{output} (expected deliverable), plus \textit{formula} and \textit{depends\_on} fields.

\textbf{V (Validation):} Set of criteria for verifying whether results achieve the goal. Systematically checks Goal achievement, Constraint compliance, Formula accuracy, and Assumption maintenance.

\subsection{Component Classification: 4-Stage Invalidation Test}

PWBS presents a systematic criterion for classifying conditions into Goal, Assumption, Formula, and Constraint. The key question is ``If this condition changes or disappears, what is invalidated?''

\begin{table}[h]
\centering
\caption{4-Stage Invalidation Test.}
\label{tab:invalidation}
\begin{tabular}{lll}
\toprule
\textbf{If changed\ldots} & \textbf{What is invalidated} & \textbf{Classification} \\
\midrule
The task itself changes & The question itself & \textsc{Goal} \\
Reasoning process invalidated & Logical development & \textsc{Assumption} \\
Calculation results change & Numerical outputs & \textsc{Formula} \\
Solution becomes unsuitable & Solution eligibility & \textsc{Constraint} \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Examples.} In code debugging, \texttt{buggy\_code} changes $\rightarrow$ different task $\rightarrow$ \textsc{Goal}; ``input is int type'' false $\rightarrow$ analysis invalid $\rightarrow$ \textsc{Assumption}; ``existing tests pass'' violated $\rightarrow$ answer unsuitable $\rightarrow$ \textsc{Constraint}. In logical reasoning with formulas, ``score = exam$\times$0.4 + \ldots'' formula changes $\rightarrow$ calculation results differ $\rightarrow$ \textsc{Formula}.

\textbf{Dual Registration.} The same condition can appear in both \textsc{Goal} (directionality) and \textsc{Constraint} (verifiable condition). Based on the engineering principle that excessive explicitness is safer than omission.

\subsection{Component Relationships}

The six components form interdependent relationships: Assumptions are preconditions for Tasks; Formulas are calculation rules for Tasks; Constraints are boundary conditions for Tasks; Validation references all of Goal, Assumption, Formula, and Constraint.

\subsection{Design Rationale}

\subsubsection{Why Separate Assumption and Constraint from Goal?}

Three structural advantages: (1) \textbf{Explicit Attention Allocation}---separated elements become independent checkpoints for LLM attention, addressing lost-in-the-middle phenomena \citep{liu2024lost}; (2) \textbf{Change Impact Tracing}---enables immediate tracking of which Tasks are affected by Assumption changes; (3) \textbf{Multi-dimensional Validation}---enables independent verification of Goal, Assumption, Formula, and Constraint compliance.

\subsubsection{Positioning: CoT Enhancement Framework}

PWBS does not replace CoT. It strengthens CoT by adding a structuring layer:

\begin{itemize}[leftmargin=2em]
  \item CoT: ``Think step by step''
  \item CoT + A: ``State premises first, then think step by step''
  \item CoT + C: ``Check constraints while thinking step by step''
  \item CoT + F: ``Apply formulas precisely while thinking step by step''
  \item CoT + V: ``Think step by step, then verify''
  \item CoT + ACFV (PWBS): All combined
\end{itemize}

Experiments confirmed that selective application of components matching task and model is more effective than applying all components.

\subsection{Adaptive Tolerance}

A paradox was observed in pilot experiments: PWBS, which better recognizes constraints, showed lower success rates. In travel planning, PWBS's initial success rate was only 20\%. This ``Constraint Rigidity Paradox'' occurs because PWBS faithfully attempts to meet strict constraints, failing to find feasible solutions, while Direct/CoT loosely interprets constraints and succeeds through uncontrolled relaxation.

Adaptive Tolerance resolves this with ``strict first, relax only on failure'':

\begin{table}[h]
\centering
\caption{Adaptive Tolerance mechanism.}
\label{tab:tolerance}
\begin{tabular}{lll}
\toprule
\textbf{Step} & \textbf{Constraint Value} & \textbf{Action} \\
\midrule
Step 0 & Original (e.g., budget \$5,000) & Execute $\rightarrow$ check \\
Step 1 & +10\% relaxed (\$5,500) & Only if failed \\
Step 2 & +20\% relaxed (\$6,000) & Only if failed \\
Step 3 & +30\% relaxed (\$6,500) & Final result \\
\bottomrule
\end{tabular}
\end{table}

Relaxation direction is automatically determined by constraint type:

\begin{table}[h]
\centering
\caption{Constraint type classification.}
\label{tab:constraint-types}
\begin{tabular}{lll}
\toprule
\textbf{Type} & \textbf{Direction} & \textbf{Example} \\
\midrule
\texttt{upper\_bound} & +10\% increase & Budget 500K $\rightarrow$ 550K \\
\texttt{lower\_bound} & $-$10\% decrease & ROE 10\% $\rightarrow$ 9\% \\
\texttt{discrete} & No relaxation & Test pass/fail \\
\bottomrule
\end{tabular}
\end{table}

\subsection{Constraint Coupling Classification}

Another unexpected finding: ``Structuring is not always beneficial.'' In travel planning, fine-grained Task decomposition caused success rate to drop from 100\% to 60\%.

The cause was interdependent constraints. Budget constraints are the sum of accommodation, food, and transport; daily travel time depends on visit order. Decomposing these into separate Tasks prevents global optimization.

\midrule
Coupled & Total + sub-items coexist & Unified Task & Travel \\
Pipeline & Sequential stages & Separate Tasks & Logic, code \\
Independent & Independent constraints & Default decomposition & -- \\
\bottomrule
\end{tabular}
\end{table}

\section{Method}

\subsection{PWBS Prompt Template}

The PWBS template structures prompts as follows:

\begin{Verbatim}[frame=single,fontsize=\small]
[GOAL]
{Ultimate objective, measurable and unambiguous.
 Include original data/code.}

[ASSUMPTIONS]
- A1: {First assumption}
  Impact if false: {Impact specification}

[FORMULA]
(When applicable)
- F1: {Formula verbatim -- do not abstract}
  Applicable to: {Which Tasks}

[CONSTRAINTS]
- C1: {First constraint}
  Impact if violated: {Impact specification}

[TASKS]
- T1:
    why: {Rationale}
    method: {Approach}
    output: {Expected deliverable}
    formula: {Formula ID if applicable}
    depends_on: {Dependencies}

[VALIDATION]
- V1: Goal achievement verification
- V2: Constraint compliance verification
- V3: Formula application accuracy
- V4: Assumption maintenance verification
\end{Verbatim}

\textbf{Design principles:} (1) \textbf{Ordering}---G$\rightarrow$A$\rightarrow$F$\rightarrow$C$\rightarrow$T$\rightarrow$V reflects natural reasoning flow; (2) \textbf{Dependency Declaration}---each Task explicitly references its dependencies; (3) \textbf{Self-Documentation}---requiring ``impact if false'' for Assumptions and ``why'' for Tasks.

\subsection{Agent Architecture}

Three modules comprise the PWBS agent: \textbf{Planner} (fills PWBS template, surfaces implicit assumptions), \textbf{Executor} (executes Tasks with dependencies as context), \textbf{Validator} (verifies outputs on four dimensions: Goal, Assumption, Formula, Constraint).

\subsection{Execution Loop}

\begin{enumerate}[leftmargin=2em]
  \item \textbf{Phase 1: Planning}---Planner generates 6-tuple
  \item \textbf{Phase 2: Stepwise execution} with immediate validation
  \item \textbf{Phase 3: Integrated validation} at Goal level
  \item \textbf{Phase 4: Feedback loop}---type-specific retry on failure
\end{enumerate}

\subsection{Integration with CoT and Existing Methods}

PWBS serves as a meta-structure that integrates with existing methods. CoT can be used within each Task's method; Tree of Thoughts can be applied for branching. PWBS provides the frame specifying under what premises and constraints these techniques operate.

\subsection{AI Planner Module (PWBS Auto)}

To verify PWBS's practicality, an AI Planner module automatically converts natural language task descriptions into the 6-tuple structure.

\begin{itemize}[leftmargin=2em]
  \item \textbf{Exp 1 (PWBS manual):} Human expert writes 6-tuple $\rightarrow$ 1 call. Verifies structural effect.
  \item \textbf{Exp 2 (PWBS auto):} AI Planner generates 6-tuple (1 call) $\rightarrow$ Executor (1 call) = 2 calls. Verifies automation feasibility.
\end{itemize}

Both experiments together verify both structural effectiveness and automation practicality.

\subsubsection{Data Preserving}

AI Planner omitted original data tables in 2-call architecture, causing Executor to hallucinate data. Solution: explicit instruction to include original data in Goal section, plus automatic verification of key numerical values.

\subsubsection{Formula Preserving}

AI Planner abstracted formulas to natural language, losing coefficients. Solution: automatic extraction of formula patterns from original input, post-processing to inject original formulas when coefficients are missing.

\subsection{Reasoning Consistency Analysis}

75 tasks $\times$ 2 methods $\times$ 3 repetitions = 450 calls (temperature=0.7, GPT-4o).

\textbf{RC Calculation.} For each task, we run 3 independent trials and compute pairwise agreement. For code debugging and logical reasoning tasks with deterministic answers, RC is the proportion of trial pairs producing identical final answers (exact match after normalization). For travel planning tasks with open-ended outputs, RC is the average pairwise semantic similarity (cosine similarity of GPT-4o embeddings) among the 3 outputs, thresholded at 0.9 for ``agreement.'' Perfect Agreement (3/3) means all 3 trials agree. This metric differs from Self-Consistency \citep{wang2023selfconsistency}, which aggregates diverse paths for better answers; our RC instead \emph{measures} whether a prompting method induces stable reasoning paths.

\subsection{CSR Transparency Paradox}

\begin{table}[h]
\centering
\caption{Constraint Satisfaction Rate (CSR) measurement.}
\label{tab:csr}
\begin{tabular}{lcccc}
\toprule
\textbf{Method} & \textbf{Checkable} & \textbf{Skipped} & \textbf{Violations} & \textbf{CSR} \\
\midrule
Direct & 127 & 97 (43\%) & 20 & 84.3\% \\
CoT & 134 & 90 (40\%) & 25 & 81.3\% \\
Plan-and-Solve & 109 & 115 (51\%) & 14 & 87.2\% \\
PWBS manual & 150 & 74 (33\%) & 77 & \textbf{48.7\%} \\
PWBS auto & 149 & 75 (34\%) & 45 & \textbf{69.8\%} \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Cause: Measurement Bias.} Direct/CoT process constraints implicitly, omitting values from responses $\rightarrow$ parser cannot judge $\rightarrow$ excluded from CSR denominator $\rightarrow$ CSR overestimated. PWBS reports constraints explicitly $\rightarrow$ violations detectable $\rightarrow$ included in denominator $\rightarrow$ accurate measurement.

\textbf{PWBS's low CSR is not performance degradation but improved transparency.} High CSR in other methods reflects ``can't see violations'' rather than ``no violations.'' We acknowledge this interpretation requires further validation: a human annotation study measuring true constraint violation rates across all methods would be needed to confirm that skipped constraints in Direct/CoT are violated at comparable rates (see Limitations, Section~6.5).

Proposed corrections: \textbf{Adjusted CSR} = Satisfied / Total Constraints (including skips); \textbf{Detectability Rate} = Checked / Total.

\section{Discussion}

\subsection{Failure-Driven Evolution}
\label{sec:evolution}

PWBS evolved through iterative failure analysis, not theoretical design. The following table documents the evolution:

\begin{table}[h]
\centering
\caption{PWBS failure-driven evolution.}
\label{tab:evolution}
\small
\begin{tabular}{clllll}
\toprule
\textbf{Phase} & \textbf{Version} & \textbf{Failure} & \textbf{Root Cause} & \textbf{Solution} & \textbf{Result} \\
\midrule
1 & v0.1$\to$v0.2 & Travel 20\% & Rigidity Paradox & Adaptive Tolerance & +40\%p \\
2 & v0.2$\to$v0.3 & Data fabrication & 2-call data loss & Data Preserving & Logic success \\
3 & v0.3$\to$v0.4 & Formula abstract. & No Formula comp. & FORMULA (6-tuple) & Formula success \\
4 & v0.4$\to$v0.5 & Travel 100$\to$60\% & Over-decomposition & Constraint Coupling & Overall 93.3\% \\
5 & v0.5$\to$v0.6 & CSR 48.7\% & Measurement bias & Adjusted CSR & Fair comparison \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Phase 1---Constraint Rigidity Paradox.} Initial 5-tuple showed only 20\% success in travel planning. PWBS faithfully tried to meet strict constraints and failed, while Direct loosely interpreted them. Adaptive Tolerance replaces uncontrolled relaxation with systematically controlled relaxation.

\textbf{Phase 2---Data Loss in 2-call Architecture.} AI Planner omitted data tables, causing Executor to fabricate data. Similar to misapplied information hiding in software engineering---procedures were correctly encapsulated but data was also hidden.

\textbf{Phase 3---Formula Abstraction (5-tuple $\to$ 6-tuple).} Formulas abstracted to natural language, losing coefficients. No existing component could properly house formulas: Goal mixes with task definition, Assumption is semantically mismatched, Constraint differs in nature, Task.method treated as abstraction target. This analysis made Formula's separation as an independent component inevitable.

\textbf{Phase 4---Over-Decomposition.} After 6-tuple introduction, logic/code reached 100\% but travel dropped from 100\% to 60\%. Travel's budget constraint is a sum of sub-items; decomposing into separate Tasks prevents global optimization. Key lesson: ``Structuring is not always beneficial.''

\textbf{Phase 5---CSR Transparency Paradox.} PWBS CSR of 48.7\% vs Direct's 84.3\% was not a performance difference but measurement bias from PWBS's higher constraint detectability.

\subsection{Repositioning PWBS as an Adaptive Toolkit}

The most important finding: \textbf{PWBS should be used as an adaptive toolkit}, not a monolithic framework.

\begin{table}[h]
\centering
\caption{Adaptive component selection guide.}
\label{tab:guide}
\begin{tabular}{lll}
\toprule
\textbf{Context} & \textbf{Recommended Config} & \textbf{Rationale} \\

\subsection{Universal Value of Assumption Surfacing}

Assumption surfacing is the only component showing consistent +4.0\%p improvement regardless of model. This suggests PWBS's core contribution is not complex structuring but making explicit the assumptions LLMs implicitly create.

While CoT structures ``how to reason,'' Assumption surfacing structures ``on what basis to reason.'' This finding suggests assumption articulation is the most cost-effective intervention in prompt engineering.

\subsection{Data/Formula Inclusion: General Principle for Multi-call Architectures}

PWBS auto's Data Preserving and Formula Preserving failures reveal a general principle: \textbf{structural decomposition must not abstract away original problem data}. This principle---information fidelity---applies to all multi-call prompting systems, not just PWBS.

\subsection{Limitations}

\begin{enumerate}[leftmargin=2em]
  \item \textbf{Task type dependency:} Public benchmark results (GSM8K \citep{cobbe2021gsm8k}, BBH \citep{suzgun2023challenging}) confirm PWBS is most effective for multi-constraint structured outputs, not single-answer problems. This scope was identified \emph{post-hoc} through experimental results rather than pre-registered as a hypothesis, which limits the strength of the claim. Applicability to other task types (e.g., open-ended generation, dialogue) remains unexplored. Future work should pre-register PWBS's applicability scope and validate on domain-specific benchmarks (e.g., TravelPlanner, SWE-bench Lite) that require structured outputs.

  \item \textbf{Model diversity:} Limited to GPT-4o and GPT-4o-mini. Generalization to Claude, Llama, etc. requires verification.

  \item \textbf{Lack of statistical significance:} No significant SR differences between methods on self-designed benchmarks ($n=75$), possibly due to benchmark size. Ablation observations ($n=25$ per category) are directional and underpowered for formal significance testing.

  \item \textbf{Evaluator bias (LLM-as-judge):} GPT-4o serves as both task executor and judge evaluator, creating potential self-preference bias \citep{zheng2024judging}. The same model may systematically favor its own output style or reasoning patterns. While public benchmark evaluation uses objective exact matching (mitigating this concern for GSM8K/BBH), the self-designed benchmark evaluation relies on GPT-4o judge scoring for Task Success Rate and Step Validity Rate. A partial human annotation study (e.g., 10--20\% sample) would strengthen these results; we leave this for future work.

  \item \textbf{Human dependency of PWBS manual:} Requires expert 6-tuple authoring. PWBS auto mitigates this at 2-call cost.

  \item \textbf{2-call overhead:} PWBS auto's Planner adds latency and cost. For simple tasks, this overhead outweighs structural benefits, as demonstrated by public benchmark results.

  \item \textbf{CSR Transparency Paradox requires further validation:} Our claim that PWBS's low CSR (48.7\%) reflects improved transparency rather than worse performance is based on the observation that other methods have higher skip rates (43\% vs 33\%). However, we lack direct evidence that skipped constraints were actually violated at similar rates. A human evaluation study measuring true constraint violation rates across all methods would be needed to substantiate this claim fully.
\end{enumerate}

\section{Conclusion}

\subsection{Summary}

We proposed PWBS, applying WBS principles to LLM prompt design with a 6-tuple (G-A-F-C-T-V) and systematic 4-stage Invalidation Test.

Key findings from 75 self-designed and 210 public benchmark tasks:

\begin{enumerate}[leftmargin=2em]
  \item Assumption surfacing yields +4.0\%p model-agnostic improvement.
  \item PWBS effectiveness inversely correlates with model capability.
  \item Component accumulation shows diminishing returns; selective application is superior.
  \item Code debugging Reasoning Consistency improves +32.0\%p.
  \item CSR Transparency Paradox: PWBS's low CSR reflects improved transparency.
  \item Public benchmarks confirm PWBS is most effective for structured-output tasks (boolean\_expressions: 96.7\%), while single-answer tasks show limited benefit.
\end{enumerate}

\subsection{Redefined Contributions}

\begin{itemize}[leftmargin=2em]
  \item Discovery of universal value of Assumption surfacing
  \item Identification of inverse model capability--structure demand relationship
  \item PWBS as adaptive toolkit with selective application guide
  \item Transparent record of failure-driven evolution (5-tuple $\to$ 6-tuple)
  \item Multi-call information preservation principles (Data/Formula Preserving)
\end{itemize}

\subsection{Future Work}

\begin{enumerate}[leftmargin=2em]
  \item Domain-specific public benchmarks (TravelPlanner, SWE-bench Lite) targeting multi-constraint structured outputs
  \item Model diversity (Claude, Llama, Gemini)
  \item Automatic component selection via meta-learner
  \item Independent study of Assumption surfacing across prompting paradigms
  \item Task-type classifier to predict PWBS applicability before deployment
\end{enumerate}

\subsection{Practical Recommendations}

\begin{itemize}[leftmargin=2em]
  \item \textbf{Always state Assumptions:} highest cost-effectiveness intervention.
  \item \textbf{Add Constraints for weak models:} compensates unrecognized constraints.
  \item \textbf{Minimize structure for strong models:} excessive structure disperses attention.
  \item \textbf{Use Formula when formulas present:} prevents coefficient loss.
  \item \textbf{Don't separate coupled constraints:} hinders global optimization.
  \item \textbf{Use full PWBS when consistency matters:} structural scaffolding stabilizes reasoning paths.
\end{itemize}
